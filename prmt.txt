Feature Flag Client SDK for Node.js

Act as a Principal Software Architect and Senior Node.js Engineer.

Design and implement a reusable Node.js/TypeScript Feature Flag Client SDK.

Business Context

We have an AWS Lambda API that exposes feature flags stored in AWS AppConfig.

Application teams should not directly call the API.

Instead they should install/import a reusable SDK and use:

await featureFlagClient.fetchFlags();

await featureFlagClient.fetchFlags([
    "featureA",
    "featureB"
]);

await featureFlagClient.getFlag("featureA");

Functional Requirements

SDK Initialization

Application should initialize SDK during startup.

Example:

FeatureFlagClient.initialize({
    clientId: process.env.CLIENT_ID,
    clientSecret: process.env.CLIENT_SECRET,
    endpoint: "https://api.company.com/feature-flags",
    pollingIntervalSeconds: 120,
    defaultValues: {
        featureA: false,
        featureB: true
    }
});

Authentication

SDK should support client credentials.

Client ID and Client Secret will be configured by consuming application.

During initialization:

1. Authenticate with backend API.
2. Obtain access token.
3. Store token securely in memory.
4. Automatically refresh token if expired.
5. Retry token acquisition on failures.

Fetch All Flags

const flags = await featureFlagClient.fetchFlags();

Response:

{
  "featureA": true,
  "featureB": false,
  "configA": {
      "timeout": 30,
      "region": "ap-south-1"
  }
}

Fetch Selected Flags

const flags = await featureFlagClient.fetchFlags([
    "featureA",
    "configA"
]);

Response:

{
  "featureA": true,
  "configA": {
      "timeout": 30,
      "region": "ap-south-1"
  }
}

Single Flag Fetch

const value = await featureFlagClient.getFlag("featureA");

Response:

true

Caching Requirements

In-Memory Cache

SDK should maintain an in-memory cache.

Example:

Map<string, any>

or equivalent thread-safe structure.

Startup Behavior

When application starts:

1. Cache is empty.
2. SDK should immediately call backend API.
3. Populate cache.
4. If API unavailable:
   - Return configured default values.
   - Continue retrying in background.

Runtime Behavior

Whenever application requests flags:

1. Check cache first.
2. If cache contains value:
   - Return cached value.
3. If cache missing:
   - Call backend API.
   - Update cache.
   - Return result.
4. If API fails:
   - Return default value if available.
   - Never crash application.

Polling / Synchronization

SDK should support background polling.

Default:

120 seconds

Configurable:

pollingIntervalSeconds

Background scheduler should:

1. Call API periodically.
2. Fetch latest flags.
3. Update cache.
4. Replace stale values.
5. Log sync results.

Reliability Requirements

Implement:

Retry

Use exponential backoff.

Example:

1 sec
2 sec
4 sec
8 sec

Max retry configurable.

Timeout

API timeout configurable.

Default:

5 seconds

Circuit Breaker

If API repeatedly fails:

1. Open circuit.
2. Serve cached data.
3. Retry after cool-down period.

Graceful Degradation

Always prefer:

Cache
→ Default Values
→ Empty Response

Never crash consuming application.
