API Service Pattern (Principal Developer Portal Marketplace)
🧭 1. Purpose

The API Service Pattern is a reusable template that enables teams to quickly create and deploy new backend API services on Principal Cloud using AWS serverless technologies.
It is not a standalone application — it is a base pattern repository registered in the Principal Developer Portal (PDP) marketplace.

When a team selects this pattern in the PDP Portal, a new GitHub repository is automatically generated with all required configurations, infrastructure code, and CI/CD pipelines prebuilt.

🧩 3. Architecture Summary

Key Components:

AWS Lambda Functions: Handle API requests, event listeners, and background processing.

Amazon API Gateway: Exposes REST endpoints for Lambda integrations.

DynamoDB: Persists data and emits streams for event propagation.

EventBridge: Central event bus for publishing and consuming domain events.

AppConfig: Manages runtime feature flags.

CDK Constructs: Shared infrastructure components (API, storage, permissions) imported from the constructs repository.

GitHub Actions: Automates build, test, and deployment workflows.

Design Principles:

Serverless-first, event-driven architecture.

Zero manual configuration for developers.

Secure and compliant by design (IAM policies, encryption, observability included).

Self-contained template for easy reuse and governance.

⚙️ 5. Lambda Handlers — Summary
Handler	Description
health-check-handler.ts	Checks application and DynamoDB health; used for ALB and monitoring probes.
example-api-service-handler.ts	Demo API showing Principal logging (pfLogger) and AWS X-Ray tracing.
event-listener-handler.ts	Handles SNS/EventBridge messages; used as an event-driven consumer template.

failover-check-handler.ts	(To confirm) Checks failover or DR status of service deployment.
🏗️ 6. Infrastructure Overview

The infrastructure code under /infrastructure defines AWS resources using AWS CDK (TypeScript).
However, this pattern does not create raw resources directly — it imports Principal’s shared CDK construct libraries from a separate repository.

Example (Conceptual)
import { ApiServiceConstruct } from '@principal/constructs';
new ApiServiceConstruct(this, 'ApiService', {
  handlers: ['health-check', 'example-api-service'],
  enableDynamoDB: true,
  enableEventBridge: true
});


Infra Components Created (via constructs):

Lambda functions (one per handler)

API Gateway routes

DynamoDB tables with streams

EventBridge event bus and rules

IAM roles and permissions

Optional AppConfig and S3 resources

🔄 7. CI/CD Workflow (GitHub Actions)
Step	Description
Build	npm ci, npm run build, linting, and unit tests.
CDK Synth	Generates CloudFormation templates via cdk synth.
Deploy	Deploys stack via cdk deploy --require-approval never.
Post-Deploy Tests	Executes e2e tests to verify API health and event flow.

Trigger Points:

Pull Request: Build + Test workflow.

Merge to Main: Full deploy to environment (dev/test/prod).

Manual trigger (optional): For redeploys or environment promotions.

Secrets Required:

AWS_ACCESS_KEY_ID

AWS_SECRET_ACCESS_KEY

AWS_REGION

Optional: CDK_CONTEXT, GITHUB_TOKEN

🌐 8. Pattern Lifecycle in PDP Portal
Step-by-Step Flow

Pattern Registration:
Platform team maintains and registers this repo in the PDP Portal Marketplace.

Pattern Selection:
Any team visits the PDP Portal → selects API Service Pattern.

New Repo Generation:
PDP automatically creates a new GitHub repo (e.g., team-order-api) using this pattern as the base.

Code Customization:
The new team adds business logic, endpoints, or event integrations.

Deployment:
CI/CD pipelines deploy the service using CDK constructs automatically.

Maintenance:
Platform team updates the pattern (e.g., v1.1 → v1.2).
New teams get updates by rebasing or reinitializing pattern versions.

⚠️ 10. Risks & Guardrails
Risk	Impact	Mitigation
SNS / DynamoDB recreation	ARN changes break Lambda subscriptions	Use RemovalPolicy.RETAIN and stable logical IDs
Construct version drift	Pattern breaks when construct versions change	Pin construct versions in package.json
Infra dependency ordering	Lambda deployed before SNS/EventBridge	Enforce deploy sequence and use outputs
Duplicate event processing	DynamoDB stream replay	Implement idempotency keys in Outbox pattern
AppConfig latency	Delay fetching feature flags	Cache or preload configuration at cold start
🚀 11. Developer Onboarding (for Teams Using This Pattern)

Validate /health endpoint and e2e tests.

Register new service in monitoring systems.
